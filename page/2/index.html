[js][DOM] JS 在 <head> 與 <body> 標籤中運行的差異

在處理 js 的程式中常常會有很多疑惑，js 常常在某些條件下能執行，
在某一些條件下卻又莫名的掛掉了，讓我很不解阿 =”=

尤其是 js 的程式有時候能在 中執行，但有些條件下又不行了…???
直到看了 Kuro Hsu 大大的文章才解惑阿…。

其實 js 的執行情況跟 瀏覽器中解析的 DOM 機制有很大的關西，一般 JS 放置的位置不外乎就兩個地方，head 與 body 標籤中。

  • head 標籤裡
  • body 標籤裡

但其實這兩個位置在運作上就有很大的差異，如下的範例:

  • 程式以 document.querySelector 取得 id=”A” 的節點，並使用文字內容 textContent 把 ‘HELLO’ 的字串放在id=”A” 的節點裡面為例

情況:

  • 原本可以在 body 內執行的 script 標籤程式，換到 head中卻無法執行。
  • 在 head 的 script 無法操作 body 中的節點，

原因:

  • 原來是因為 網頁被載入瀏覽器後，瀏覽器會由上而下 來解析該檔。
  • 而在 head 標籤中解析到 script 時，會立即執行 script 的內容，
  • 但 body內容卻還沒被解析，自然無法找到 body內的節點。

了解完之後猶如醍醐灌頂，原來 撰寫 JS 程式真要能熟練，可不只是要知道JS核心 的物件導向語法、還得對 瀏覽器中的 BOM tree 模型與BOM 的核心 window 物件有一定的了解才行阿…。

[laravel] [作品集1] 以 migration資料遷移程式 新增db中的資料表

規劃設計一個能有左選單樹進行分類的作品集介面，如下圖:

此章節先從新增資料庫設計開始，我習慣先設計資料庫的結構與欄位，如下圖。
因為每個作品都有其分類，自然在作品表中需要對應一個分類id，portfolio_category_id 。
既然作品有分類，當然就需要另外設計一個作品分類表，畢竟設計是樹狀選單，使用無線層的樹狀結構設計最常見。
也就是每個分類再一對一對應其父分類的id，用遞迴就能簡單長出整棵樹了。

先下指令新增 作品 與 作品分類的 migration 資料遷移程式

php artisan make:migration create_portfolio_table
php artisan make:migration create_portfolio_category_table

下完新增 migration 的資料遷移程式後，便會在 laravel的 [根目錄] /database/migrations/ 目錄下新增程式，如下圖:

點開如上圖的程式，可以看到剛新增的 migration資料遷移程式中，會有個 up() 的函式，
使用 Schema 類的 create()方法在新增表格時定義其欄位，
通常需要修改的就是 Schema::create( … ) 中的定義欄位內容。

public function up()
{
    Schema::create('portfolio', function (Blueprint $table) {
        $table->id();
        $table->timestamps();
    });
}

依照之前的欄位設計，修該兩之程式的 Schema::create( … ) 中的定義欄位內容，如下:

  • portfolio 作品表
public function up()
{
    Schema::create('portfolio', function (Blueprint $table) {
        $table->bigIncrements('id');
        $table->bigInteger('portfolio_category_id');
        $table->string('name');
        $table->string('discription');
        $table->bigInteger('sort');
        $table->boolean('is_show')->default(1);
        $table->boolean('is_delete')->default(0);
        $table->timestamps();
    });
}
  • portfolio_category 作品分類表
public function up()
{
    Schema::create('portfolio_category', function (Blueprint $table) {
        $table->bigIncrements('id');
        $table->bigInteger('parent_id');
        $table->string('name');
        $table->bigInteger('sort');
        $table->boolean('is_show')->default(1);
        $table->boolean('is_delete')->default(0);
        $table->timestamps();
    });
}

最後下資料遷移的指令，就能將migration 程式中的欄位定義 patch到資料庫: 

php artisan migrate

成功執行畫面:

資料庫也成功新增此兩表:

到這邊就成功的使用 migration 資料遷移程式新增資料表與定義欄位了，
下一章將會一樣使用 migration 資料遷移程式在此兩表插入測試資料。

[編碼] 關於\r 與\n 字符

關於 \r\n 介紹

  • \r\n通常是微軟的文件、pdf 會產生的，在其它的編輯器裡面，

  • 會在一行的節尾看到 ^M，那就是\r\n

  • 是\r\n，不是\n\r ，會無效

    字符 \r 與 \n 差異

  • ‘\r’是回車( carriage return) ，’\n’是換行 ( line feed )

  • ‘\r’ 是光標到行首，’\n’是光標下移一行

“回車”（carriage return） 由來

  • 來自打字機 與 電傳打字機（Teletype Model 33）

換行符號有兩種，一個叫\r : Cr(carriage return)，一個是\n : Lf(line feed)

  • Cr是將指標移到最前頭（回車）
  • Lf是跳到下一行的位置，但沒有跳到開頭的效果

不同的作業系統，其換行符規則也不同:

  • Mac只有 \r Cr
  • Linux/Unix只有\n Lf
  • Windows則是CrLf，兩者皆有，\r\n
